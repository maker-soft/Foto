# Полная инструкция по развёртыванию сайта Лены Сибирской на GitHub Pages и Supabase

## 1. Что вы получите после настройки

После выполнения инструкции будут работать два адреса:

- публичный сайт: `https://ВАШ_ЛОГИН.github.io/ИМЯ_РЕПОЗИТОРИЯ/`;
- закрытая админка: `https://ВАШ_ЛОГИН.github.io/ИМЯ_РЕПОЗИТОРИЯ/admin/`.

Ссылки на админку в публичной части сайта нет. Адрес `/admin/` необходимо сохранить отдельно. Вход выполняется по паролю, который хранится и проверяется в Supabase Auth. Тексты сайта хранятся в таблице Supabase, фотографии — в Supabase Storage.

> Важно: GitHub Pages публикует статические файлы открыто. Скрытый адрес админки не является защитой сам по себе. Реальная защита обеспечивается входом через Supabase Auth и политиками RLS.

---

## 2. Что потребуется

1. Аккаунт GitHub.
2. Аккаунт Supabase.
3. Распакованный архив сайта.
4. Email администратора — лучше отдельный рабочий адрес.
5. Сложный пароль администратора длиной не менее 12 символов.
6. Текстовый редактор: Блокнот, Notepad++, VS Code или аналогичный.

Необходимые файлы проекта:

| Файл | Для чего нужен | Что в нём менять |
|---|---|---|
| `assets/js/config.js` | Подключение сайта к Supabase | URL проекта, публичный ключ, email администратора |
| `supabase/schema.sql` | Создание таблицы, хранилища и прав | Заменить `admin@example.com` на email администратора |
| `assets/js/default-content.js` | Начальное содержимое до первого сохранения | Обычно не менять; редактировать сайт через админку |
| `index.html` | Публичная часть сайта | Обычно не менять |
| `admin/index.html` | Страница входа и интерфейс админки | Обычно не менять |

# Часть I. Настройка Supabase с нуля

## 3. Создание аккаунта и проекта Supabase

1. Откройте сайт Supabase и войдите в аккаунт.
2. Нажмите **New project**.
3. Выберите существующую организацию или создайте новую.
4. Заполните поля:
   - **Name** — например, `lena-sibirskaya-site`;
   - **Database Password** — отдельный сложный пароль базы данных;
   - **Region** — ближайший к основной аудитории регион;
   - тариф — для старта достаточно доступного бесплатного варианта, если его ограничения подходят.
5. Сохраните пароль базы данных в надёжном месте. Он не вставляется в сайт, но потребуется для административных операций и восстановления.
6. Нажмите **Create new project** и дождитесь окончания создания.

Пароль базы данных и пароль входа в админку — это два разных пароля.

---

## 4. Определите email и пароль администратора

Пример:

```text
Email администратора: photo-admin@example.com
Пароль администратора: ваш сложный пароль
```

Email должен быть одинаковым в трёх местах:

1. пользователь в **Supabase Authentication**;
2. параметр `ADMIN_EMAIL` в `assets/js/config.js`;
3. все проверки email в `supabase/schema.sql`.

Рекомендуется использовать email в нижнем регистре без пробелов.

---

## 5. Создание пользователя-администратора

1. В проекте Supabase откройте **Authentication → Users**.
2. Нажмите **Add user**.
3. Выберите создание нового пользователя с email и паролем. Название пункта может отображаться как **Create new user**.
4. Введите выбранный email администратора.
5. Задайте сложный пароль.
6. Включите подтверждение пользователя / **Auto Confirm User**, если этот переключатель отображается.
7. Сохраните пользователя.
8. Убедитесь, что у пользователя статус подтверждён и поле подтверждения email не пустое.

Если в интерфейсе доступно только приглашение по email, используйте штатное создание пользователя с паролем в панели Authentication. Не вставляйте `service_role` или secret key в HTML для создания пользователя.

После создания администратора рекомендуется открыть **Authentication → Providers → Email** и отключить возможность самостоятельной регистрации новых пользователей, оставив вход существующего администратора. Параметр обычно называется **Allow new users to sign up**. На работу уже созданного администратора это не влияет.

---

## 6. Подготовка SQL-файла

1. На компьютере откройте файл:

```text
supabase/schema.sql
```

2. Найдите значение:

```text
admin@example.com
```

3. Выполните **Заменить всё** и вставьте фактический email администратора.

Например:

```sql
(auth.jwt() ->> 'email') = 'photo-admin@example.com'
```

В текущем файле строка `admin@example.com` встречается 9 раз, включая комментарий. Проще выполнить замену во всём файле.

4. Сохраните файл в кодировке UTF-8.

Не изменяйте следующие значения, если не планируете синхронно менять код сайта:

```text
CONTENT_ROW_ID = main
STORAGE_BUCKET = site-media
```

---

## 7. Выполнение SQL в Supabase

1. В Supabase откройте **SQL Editor**.
2. Нажмите **New query**.
3. Скопируйте всё содержимое изменённого файла `supabase/schema.sql`.
4. Вставьте SQL в редактор.
5. Нажмите **Run**.
6. Дождитесь сообщения об успешном выполнении.

Скрипт выполняет следующие действия:

- создаёт таблицу `public.site_content`;
- включает Row Level Security;
- разрешает всем посетителям читать опубликованное содержимое;
- разрешает добавлять, изменять и удалять содержимое только администратору с указанным email;
- создаёт публичный Storage bucket `site-media`;
- разрешает посетителям просматривать изображения;
- разрешает загружать, заменять и удалять изображения только администратору.

Скрипт можно повторно выполнить: он удаляет и создаёт политики заново, не удаляя опубликованное содержимое.

---

## 8. Проверка созданных объектов Supabase

### 8.1. Таблица

1. Откройте **Table Editor**.
2. Найдите таблицу `site_content`.
3. До первого сохранения из админки таблица может быть пустой — это нормально.
4. После первого сохранения появится строка:

```text
id = main
```

### 8.2. Хранилище

1. Откройте **Storage**.
2. Найдите bucket `site-media`.
3. Убедитесь, что он отмечен как публичный.

### 8.3. Политики RLS

Проверьте политики таблицы `site_content` и объекта `storage.objects`. Для записи должны использоваться условия с email администратора.

---

## 9. Получение URL проекта и публичного ключа

1. Откройте в Supabase **Settings → API Keys** либо окно **Connect** проекта.
2. Скопируйте **Project URL**. Он выглядит примерно так:

```text
https://abcdefghijk.supabase.co
```

3. Скопируйте **Publishable key**. Новый ключ обычно начинается с:

```text
sb_publishable_
```

В старых проектах может отображаться legacy-ключ `anon`. Для этого сайта подходит Publishable key или legacy `anon` key.

> Никогда не используйте в браузере Secret key или `service_role`. Они обходят RLS и дают повышенные права.

# Часть II. Изменение файлов сайта

## 10. Заполнение `assets/js/config.js`

Откройте файл:

```text
assets/js/config.js
```

Исходный вариант:

```javascript
window.APP_CONFIG = {
  SUPABASE_URL: "https://YOUR_PROJECT.supabase.co",
  SUPABASE_ANON_KEY: "YOUR_SUPABASE_ANON_KEY",
  ADMIN_EMAIL: "admin@example.com",
  CONTENT_ROW_ID: "main",
  STORAGE_BUCKET: "site-media"
};
```

Замените только первые три значения.

Пример заполнения:

```javascript
window.APP_CONFIG = {
  SUPABASE_URL: "https://abcdefghijk.supabase.co",
  SUPABASE_ANON_KEY: "sb_publishable_ВАШ_ПУБЛИЧНЫЙ_КЛЮЧ",
  ADMIN_EMAIL: "photo-admin@example.com",
  CONTENT_ROW_ID: "main",
  STORAGE_BUCKET: "site-media"
};
```

Несмотря на имя переменной `SUPABASE_ANON_KEY`, в неё можно и рекомендуется вставить современный **Publishable key**. Название переменной сохранено для совместимости кода.

Проверьте:

- кавычки не удалены;
- после каждой строки, кроме последней, стоит запятая;
- email совпадает с пользователем Supabase и email в SQL;
- в файле отсутствуют `service_role`, Secret key и пароль администратора.

Пароль администратора ни в один файл не вставляется.

---

## 11. Какие файлы обычно не нужно изменять

### `assets/js/default-content.js`

Содержит стартовые тексты и структуру сайта. После подключения Supabase редактируйте данные через админку. Изменения этого файла не заменят уже сохранённую строку `main` в Supabase.

### `assets/js/backend.js`

Отвечает за авторизацию, чтение/запись данных и загрузку изображений. Без необходимости не изменяйте.

### `assets/js/site.js`

Отвечает за показ публичного сайта. Без необходимости не изменяйте.

### `admin/admin.js`

Отвечает за интерфейс админки. Пароль в нём отсутствует.

# Часть III. Локальная проверка

## 12. Почему нельзя полноценно проверять двойным кликом

При открытии `index.html` через адрес `file://` браузер может ограничивать JavaScript, запросы к Supabase и загрузку модулей. Для проверки используйте локальный HTTP-сервер.

## 13. Запуск локального сервера через Python

1. Откройте терминал или командную строку в корневой папке проекта — там, где находится `index.html`.
2. Выполните:

```bash
python -m http.server 8080
```

Если команда `python` не найдена, попробуйте:

```bash
py -m http.server 8080
```

3. Откройте в браузере:

```text
http://localhost:8080/
```

4. Админка:

```text
http://localhost:8080/admin/
```

5. Введите пароль созданного пользователя Supabase.

## 14. Что проверить локально

- главная страница загружается без предупреждения «Supabase не настроен»;
- открываются разделы школы и детского сада;
- работает меню на компьютере и в режиме мобильного экрана;
- открывается `/admin/`;
- неверный пароль не даёт войти;
- правильный пароль открывает админку;
- после изменения текста кнопка **Сохранить на сайте** обновляет публичную часть;
- фотографии загружаются в Storage;
- после обновления страницы данные сохраняются.

После проверки остановите сервер сочетанием `Ctrl+C`.

# Часть IV. Публикация на GitHub Pages

## 15. Создание репозитория GitHub

1. Войдите в GitHub.
2. Нажмите **New repository**.
3. Укажите имя, например:

```text
school-album
```

4. Самый простой вариант для GitHub Pages — публичный репозиторий.
5. Не добавляйте автоматически README, `.gitignore` и лицензию, если планируете загрузить готовое содержимое архива целиком.
6. Нажмите **Create repository**.

---

## 16. Загрузка файлов через браузер GitHub

1. Откройте созданный пустой репозиторий.
2. Нажмите **uploading an existing file** или **Add file → Upload files**.
3. Загрузите **содержимое распакованной папки**, а не сам ZIP-файл.
4. В корне репозитория должны оказаться:

```text
index.html
404.html
.nojekyll
README.md
admin/
assets/
supabase/
ИНСТРУКЦИЯ_РАЗВЕРТЫВАНИЯ.md
```

Неправильно:

```text
репозиторий/lena_sibirskaya_site_github_v2/index.html
```

Правильно:

```text
репозиторий/index.html
```

5. Внизу страницы укажите комментарий, например `Первичная публикация сайта`.
6. Нажмите **Commit changes**.

Если браузер не позволяет корректно загрузить папки, используйте GitHub Desktop или Git.

---

## 17. Включение GitHub Pages

1. В репозитории откройте **Settings**.
2. Слева выберите **Pages**.
3. В разделе **Build and deployment**:
   - **Source**: `Deploy from a branch`;
   - ветка: `main`;
   - папка: `/ (root)`.
4. Нажмите **Save**.
5. Дождитесь публикации. Обычно это занимает несколько минут.
6. GitHub покажет адрес опубликованного сайта.

Пример:

```text
https://username.github.io/school-album/
```

Админка:

```text
https://username.github.io/school-album/admin/
```

Файл `.nojekyll` добавлен, чтобы GitHub Pages публиковал статические файлы без дополнительной обработки Jekyll.

---

## 18. Первый вход после публикации

1. Откройте адрес `/admin/`.
2. Введите пароль пользователя, созданного в Supabase.
3. Email вводить не требуется: он задан в `assets/js/config.js`.
4. Измените любой тестовый текст.
5. Нажмите **Сохранить на сайте**.
6. Откройте публичный сайт в новой вкладке и обновите страницу.
7. В Table Editor Supabase проверьте появление строки `main` в `site_content`.

# Часть V. Эксплуатация и изменения

## 19. Что редактируется через админку

Через админку редактируются:

- заголовки и тексты главной страницы;
- направления «Начальная школа» и «Детский сад»;
- виды альбомов, описания, обложки и галереи;
- разделы «Подготовка» и «Процесс»;
- вопросы и ответы;
- подпапки и фотографии раздела «Фото»;
- слайд-шоу «Обо мне»;
- контакты и ссылки;
- цвета, шрифты, размеры и интервалы.

Для этих изменений повторная загрузка файлов на GitHub не нужна — данные сохраняются в Supabase.

## 20. Когда требуется обновлять GitHub

GitHub необходимо обновлять, если вы меняете:

- HTML-разметку;
- CSS-дизайн;
- JavaScript-логику;
- стартовые локальные изображения в `assets/images`;
- файл подключения `config.js`;
- структуру админки.

После замены файла на GitHub дождитесь повторной публикации Pages и обновите страницу без кеша.

---

## 21. Изменение пароля администратора

Пароль хранится только в Supabase.

1. Откройте **Authentication → Users**.
2. Найдите пользователя администратора.
3. Выполните изменение или сброс пароля доступным способом панели Supabase.
4. После смены пароля файлы GitHub менять не нужно.

## 22. Изменение email администратора

При смене email необходимо синхронно изменить:

1. email пользователя в Supabase Auth;
2. `ADMIN_EMAIL` в `assets/js/config.js`;
3. email во всех политиках `supabase/schema.sql`;
4. повторно выполнить изменённый SQL в SQL Editor;
5. загрузить обновлённый `config.js` на GitHub.

До синхронизации email вход или сохранение может не работать.

---

## 23. Резервное копирование

Рекомендуется периодически сохранять:

1. исходный ZIP-архив и актуальную копию репозитория GitHub;
2. данные строки `main` из таблицы `site_content`;
3. фотографии из bucket `site-media`;
4. файл `assets/js/config.js` без секретных ключей;
5. email администратора и пароль в менеджере паролей.

Publishable key не является секретом, но Secret key и `service_role` не должны попадать в резервные копии, доступные посторонним.

# Часть VI. Типовые ошибки

## 24. На сайте сообщение «Supabase не настроен»

Проверьте `assets/js/config.js`:

- остались `YOUR_PROJECT` или `YOUR_SUPABASE_ANON_KEY`;
- неверный Project URL;
- файл сохранён не в ту папку;
- GitHub ещё не завершил повторную публикацию.

## 25. Ошибка входа `Invalid login credentials`

Проверьте:

- правильность пароля;
- email пользователя совпадает с `ADMIN_EMAIL`;
- пользователь подтверждён;
- используется именно пароль Auth-пользователя, а не пароль базы данных;
- возможность входа по email/password включена в Authentication Providers.

## 26. Вход выполнен, но сохранить нельзя

Обычно причина — RLS:

- email в SQL отличается от email пользователя;
- SQL не выполнен или выполнился с ошибкой;
- пользователь вошёл под другим email;
- политика была изменена вручную.

Повторите замену email в `schema.sql` и снова выполните весь файл.

## 27. Ошибка `new row violates row-level security policy`

Это означает, что запрос не соответствует политике RLS. Проверьте:

- email в JWT пользователя;
- email в политике;
- наличие сессии администратора;
- политики INSERT/UPDATE для таблицы и Storage.

Не отключайте RLS как постоянное решение.

## 28. Ошибка загрузки фотографий

Проверьте:

- существует bucket `site-media`;
- bucket публичный;
- выполнены Storage policies;
- пользователь вошёл в админку;
- формат файла: JPEG, PNG, WEBP или GIF;
- размер одного файла не превышает 15 МБ;
- в `config.js` указано `STORAGE_BUCKET: "site-media"`.

## 29. GitHub Pages показывает 404

Проверьте:

- в корне репозитория есть `index.html`;
- Pages настроен на ветку `main` и папку `/(root)`;
- файлы действительно закоммичены;
- публикация завершилась без ошибки;
- открывается правильный адрес с именем репозитория.

## 30. Публичный сайт работает, а `/admin/` — нет

Проверьте наличие файла:

```text
admin/index.html
```

Адрес должен заканчиваться слешем:

```text
https://username.github.io/repository/admin/
```

## 31. После сохранения отображается старый текст

- обновите страницу сочетанием `Ctrl+F5`;
- на iPhone/Safari закройте вкладку и откройте заново;
- проверьте, что строка `main` изменилась в Supabase;
- убедитесь, что открываете тот же GitHub-сайт, который подключён к текущему Supabase-проекту.
